## 一、核心概念

LLM 应用的评估不同于传统软件测试——输出不是确定的，不能用简单的"预期 vs 实际"断言。评估需要三个层次的思考：

1. **幻觉检测**：模型是否编造了不存在的信息？输出是否与给定上下文一致？
2. **质量评估**：回答是否完整、准确、相关？是否有遗漏或冗余？
3. **A/B 对比**：两个 Prompt / 模型 / RAG 配置哪个更好？量化指标如何设计？

三者的关系：幻觉检测是底线（安全红线），质量评估是目标（用户满意），A/B 对比是迭代方法（持续优化）。

## 二、幻觉检测三层方法

### 2.1 规则层：事实校验

最简单直接的方式——从 LLM 输出中提取断言（Claim），然后与上下文做精确匹配和实体校验：

```python
import re
from typing import List, Tuple

class RuleBasedHallucinationDetector:
    """基于规则的事实一致性校验"""

    def __init__(self, context: str):
        self.context = context.lower()
        self.context_entities = self._extract_entities(context)

    @staticmethod
    def _extract_entities(text: str) -> set:
        """提取数字、日期、专有名词等关键实体"""
        numbers = set(re.findall(r'\d+(?:\.\d+)?', text))
        dates = set(re.findall(r'\d{4}-\d{2}-\d{2}', text))
        percentages = set(re.findall(r'\d+(?:\.\d+)?%', text))
        return numbers | dates | percentages

    def check(self, response: str) -> Tuple[bool, List[str]]:
        """
        校验 LLM 输出中的实体是否都在上下文中出现
        返回 (是否通过, 可疑实体列表)
        """
        response_entities = self._extract_entities(response)
        hallucinations = []

        for entity in response_entities:
            if entity not in self.context_entities:
                hallucinations.append(
                    f"实体 '{entity}' 出现在输出中但未在上下文中找到"
                )

        return len(hallucinations) == 0, hallucinations

# 使用示例
context = "2026年Q1营收为1280万元，同比增长23%。用户数达到45万。"
response = "报告显示2026年Q1营收为1280万元，同比增长23%，用户数达到50万。"

detector = RuleBasedHallucinationDetector(context)
passed, issues = detector.check(response)
print(f"通过: {passed}, 可疑点: {issues}")
# 通过: False, 可疑点: ["实体 '50' 出现在输出中但未在上下文中找到"]
```

### 2.2 模型层：用 LLM 检测 LLM

规则层只能捕获实体级别的幻觉，无法检测语义级别的编造（如"产品在海外市场表现优异"——上下文只提了国内市场）。模型层用另一个 LLM 做判断：

```python
HALLUCINATION_CHECK_PROMPT = """你是一个严格的事实核查员。请判断以下 LLM 生成的回答是否与给定的上下文一致。

## 上下文（Ground Truth）
{context}

## LLM 生成的回答
{response}

## 评估标准
- "一致"：回答中的所有事实性陈述都可以在上下文中找到依据
- "部分幻觉"：回答中有一部分陈述无法在上下文中找到依据，但核心信息正确
- "严重幻觉"：回答的核心信息与上下文严重不符或完全编造

请以 JSON 格式输出：
{{
    "verdict": "一致 | 部分幻觉 | 严重幻觉",
    "hallucinated_claims": ["编造的声明1", "编造的声明2"],
    "confidence": 0.0-1.0,
    "explanation": "简要说明"
}}
"""

def model_based_check(context: str, response: str) -> dict:
    """用 LLM 做幻觉检测（实际调用替换为你的 API）"""
    prompt = HALLUCINATION_CHECK_PROMPT.format(
        context=context, response=response
    )
    # result = llm.invoke(prompt)
    # return json.loads(result)
    return {"verdict": "一致", "hallucinated_claims": [], "confidence": 0.95}
```

### 2.3 人工层：抽样评测

自动化方法永远有漏判和误判。生产系统中应建立定期人工抽样机制：

- **频率**：每周随机抽 50 条线上对话
- **评测维度**：事实准确性（1-5）、回答完整性（1-5）、有无幻觉（是/否）
- **统计指标**：幻觉率 = 有幻觉的对话数 / 总抽样数，目标 < 5%

## 三、RAG 评估框架：RAGAS

RAGAS（RAG Assessment）是专门为 RAG 系统设计的评估框架，提供了一套标准化的指标：

```python
from datasets import Dataset
from ragas import evaluate
from ragas.metrics import (
    faithfulness,          # 忠实度：回答是否基于检索到的上下文
    answer_relevancy,      # 答案相关性
    context_recall,        # 上下文召回率
    context_precision,     # 上下文精确率
)

# 准备评测数据
eval_dataset = Dataset.from_dict({
    "question": ["2026年Q1的营收是多少？"],
    "answer": ["Q1营收为1280万元，同比增长23%"],
    "contexts": [["2026年Q1营收为1280万元，同比增长23%。用户数达到45万。"]],
    "ground_truth": ["1280万元，同比增长23%"]
})

# 评估
result = evaluate(
    eval_dataset,
    metrics=[faithfulness, answer_relevancy, context_recall, context_precision]
)
print(result)
```

**四个核心指标释义**：

| 指标 | 测量什么 | 低分意味着 |
|------|----------|-----------|
| Faithfulness | 回答是否严格基于检索到的上下文 | 模型在编造信息，幻觉严重 |
| Answer Relevancy | 回答是否切题、不跑偏 | 模型偏离了用户问题 |
| Context Recall | 检索是否找全了回答所需的上下文 | 检索丢失了关键信息 |
| Context Precision | 检索的上下文中噪声比例 | 检索了过多无关文档 |

## 四、A/B 测试框架

### 4.1 设计原则

A/B 测试不是随机分流就完事。关键设计点：

- **对照组与实验组**：改动只能有一个变量（如只换 Prompt、只换模型）
- **样本量**：每组至少 100 条有效对话，确保统计显著性
- **评测指标**：业务指标（用户满意度） + 技术指标（幻觉率、延迟、成本）
- **评测方式**：自动化指标 + 人工盲评（评测者不知道属于哪个组）

### 4.2 自动化 A/B 评测框架

```python
import json
from dataclasses import dataclass, field
from typing import List

@dataclass
class ABTestResult:
    group_a_name: str
    group_b_name: str
    metrics: dict = field(default_factory=dict)

    def report(self):
        print(f"\n{'='*50}")
        print(f"A/B 测试报告: {self.group_a_name} vs {self.group_b_name}")

        for metric, (val_a, val_b) in self.metrics.items():
            diff = val_b - val_a
            winner = "B" if diff > 0 else "A"
            direction = "↑" if diff > 0 else "↓"
            print(f"  {metric:25s}: A={val_a:.3f}  B={val_b:.3f}  "
                  f"Δ={diff:+.3f} {direction}  优胜组={winner}")

class ABTestRunner:
    def __init__(self, test_questions: List[str], ground_truths: List[str]):
        self.questions = test_questions
        self.ground_truths = ground_truths

    def run(self, model_a, model_b, evaluator_fn) -> ABTestResult:
        """执行 A/B 测试"""
        scores_a = []
        scores_b = []

        for question, truth in zip(self.questions, self.ground_truths):
            # 组 A
            answer_a = model_a(question)
            score_a = evaluator_fn(question, answer_a, truth)
            scores_a.append(score_a)

            # 组 B
            answer_b = model_b(question)
            score_b = evaluator_fn(question, answer_b, truth)
            scores_b.append(score_b)

        avg_a = sum(scores_a) / len(scores_a)
        avg_b = sum(scores_b) / len(scores_b)

        return ABTestResult(
            group_a_name="当前 Prompt",
            group_b_name="新 Prompt（含 CoT）",
            metrics={
                "平均评分": (avg_a, avg_b),
                "准确率（≥4分）": (
                    sum(1 for s in scores_a if s >= 4) / len(scores_a),
                    sum(1 for s in scores_b if s >= 4) / len(scores_b)
                )
            }
        )
```

### 4.3 评测函数实现

```python
def evaluate_answer(question: str, answer: str, ground_truth: str) -> float:
    """多维度评分（1-5）"""
    score = 0.0

    # 维度 1: 事实准确性 (40%)
    if ground_truth.lower() in answer.lower():
        score += 2.0  # 满分 2.0（40% × 5）

    # 维度 2: 完整性 (30%)
    keywords = ground_truth.split()
    matched = sum(1 for kw in keywords if kw.lower() in answer.lower())
    score += min(1.5, 1.5 * matched / max(len(keywords), 1))

    # 维度 3: 相关性 (20%)
    question_keywords = set(question.lower().split()) - {"的", "是", "在", "有"}
    relevance = sum(1 for kw in question_keywords if kw in answer.lower())
    score += min(1.0, 1.0 * relevance / max(len(question_keywords), 1))

    # 维度 4: 简洁性 (10%)
    if len(answer) < len(ground_truth) * 3:
        score += 0.5

    return score
```

## 五、最佳实践

1. **评估先行**：在开发 RAG/Agent 功能之前，先建立评估基线。用一组固定的 test set 评测当前系统，记录各项指标。之后每次迭代都跑同一组测试。

2. **指标分层**：底线指标（幻觉率 < 3%、安全合规率 100%）设为 P0 告警；质量指标（Faithfulness > 0.85、Answer Relevancy > 0.80）设为 P1 监控；体验指标（延迟、成本）设为 P2 观察。

3. **不要过度依赖 LLM-as-Judge**：用 LLM 评估 LLM 有自我偏好的问题——GPT-4 倾向于给 GPT-4 的输出打高分。应交叉使用不同厂商的 Judge 模型，并与人工抽样结果做一致性校验。

4. **A/B 测试必须盲评**：人工评测时评测者不应知道回答来自哪个组，否则潜意识偏见会扭曲结果。可以用简单脚本给每条回答随机打乱组标签。

## 面试题

### 题1：规则层和模型层的幻觉检测各有什么优缺点？为什么不能只用其中一种？

**答案**：规则层检测的优点是速度快、成本极低（无需 LLM 调用）、可解释性强（精确指出哪个实体有问题）；缺点是只能捕获**表面级别的幻觉**——实体数字不匹配、日期错误等。对于语义层面的编造（如"产品市场反应良好"而上下文根本没提市场反应），规则层无能为力。

模型层（LLM-as-Judge）的优点是能检测语义级幻觉，理解上下文的隐含信息和逻辑一致性；缺点是成本高（每次检测需要一次 LLM 调用）、有自身偏见（Judge 模型也可能犯错），以及对"灰色地带"（部分正确部分编造）的判断不稳定。

不能只用一种的原因：(1) 纯规则层会漏掉大量语义幻觉，实际幻觉率可能被严重低估；(2) 纯模型层成本随对话量线性增长，且在小样本下不稳定。推荐的生产策略是三级漏斗——规则层做全量初筛（快速过滤明显幻觉），模型层对规则层存疑的 case 做二次判定，人工抽样做第三层兜底并校准前两层的判定准确率。

### 题2：RAGAS 框架中的 Faithfulness 指标是如何计算的？它为什么是 RAG 系统最重要的指标？

**答案**：RAGAS 的 Faithfulness 计算流程分为两步：第一步，用 LLM 从生成的回答中提取一组原子级的"声明"（Claims），每个声明是一个独立的可验证陈述；第二步，对每个声明逐一检查它是否可以在检索到的上下文（Contexts）中找到支撑，统计被支撑的声明比例——Faithfulness = 被支撑的声明数 / 总声明数。

它是 RAG 最重要的指标的原因：RAG 的核心价值主张就是"基于检索到的知识回答问题而非凭空编造"。如果 Faithfulness 低，意味着模型在忽略检索结果、依赖自身参数化知识甚至幻觉来生成回答，这就完全违背了 RAG 的初衷。一个 Faithfulness 0.95 的系统意味着 95% 的回答内容都在检索到的文档中有据可查；而 0.60 的系统意味着近一半内容来自模型"脑补"——这在医疗、法律、金融等场景中是灾难性的。
