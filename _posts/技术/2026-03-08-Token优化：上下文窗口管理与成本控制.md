## 一、核心概念

AI Agent 的 Token 消耗是成本的核心变量，也是用户体验（延迟）的关键瓶颈。一个典型的 ReAct Agent 经过 5 轮思考-行动循环后，消息历史可能膨胀到 8000+ tokens——而用户原始问题可能只有 50 tokens。信息密度不足 1%。

Token 优化需要从三个维度同时发力：

- **输入侧**：压缩发送给 LLM 的上下文，减少冗余
- **缓存侧**：复用已计算的结果，避免重复推理
- **路由侧**：简单问题用小模型，复杂问题用大模型

这三个维度构成了成本控制的"铁三角"。

## 二、输入侧优化

### 2.1 Prompt 压缩（LLMLingua 实战）

不是所有 Prompt 都对答案有贡献。LLMLingua 是微软提出的 Prompt 压缩框架，通过小型语言模型评估每个 token 的信息量，保留高信息量 token、丢弃低信息量 token：

```python
from llmlingua import PromptCompressor

compressor = PromptCompressor(
    model_name="microsoft/llmlingua-2-bert-base-multilingual-cased-meetingbank",
    use_llmlingua2=True,
)

# 一个冗长的 RAG Prompt（包含大量检索到的文档片段）
long_prompt = """
你是一个专业的技术顾问。请根据以下检索到的文档内容回答用户的问题。

## 检索结果
[文档1] Python 是一种解释型、面向对象的高级编程语言，由 Guido van Rossum 
于 1991 年首次发布。Python 的设计哲学强调代码的可读性和简洁的语法...
[文档2] asyncio 是 Python 3.4 引入的标准库，提供了编写单线程并发代码的
基础设施。它使用 async/await 语法，基于事件循环（event loop）机制...
[文档3] aiohttp 是一个基于 asyncio 的异步 HTTP 客户端/服务端框架...
[文档4] Python 的 GIL（全局解释器锁）是 CPython 中的一个互斥锁...

## 用户问题
如何在 Python 中实现高性能的异步 HTTP 请求？
"""

# 压缩到原来的 50%
compressed = compressor.compress_prompt_llmlingua2(
    long_prompt,
    rate=0.5,
    force_tokens=["异步", "asyncio", "HTTP", "aiohttp"]  # 强制保留的关键词
)

compressed_text = compressed["compressed_prompt"]
original_tokens = len(long_prompt) // 4    # 粗略估算
compressed_tokens = len(compressed_text) // 4

print(f"原始 Token 估量: ~{original_tokens}")
print(f"压缩后 Token 估量: ~{compressed_tokens}")
print(f"压缩率: {(1 - compressed_tokens/original_tokens)*100:.1f}%")
```

### 2.2 消息窗口管理

ReAct 循环中，早期的消息在后期往往失去参考价值。两种策略：

**滑动窗口**（最简单）：只保留最近 N 轮对话。

```python
def trim_messages(messages: list, max_rounds: int = 3) -> list:
    """保留最近 N 轮完整交互 + 原始用户问题 + System Prompt"""
    # 保留 System Prompt 和第一条用户消息
    system_and_query = [m for m in messages
                        if m["role"] in ("system",)
                        or (m["role"] == "user" and m == messages[1])]

    # 滑动窗口：保留最近 N 轮
    recent = messages[-(max_rounds * 2):]

    return system_and_query + recent
```

**摘要窗口**（推荐）：对早期消息做摘要压缩，而非直接丢弃。

```python
def summarize_old_messages(messages: list, keep_recent: int = 3) -> list:
    """将早期消息压缩为一个摘要，保留最近几轮原文"""
    if len(messages) <= keep_recent:
        return messages

    old_messages = messages[:-keep_recent]
    recent_messages = messages[-keep_recent:]

    # 用 LLM 将旧消息压缩为一段摘要
    old_text = "\n".join([m.get("content", "") for m in old_messages])
    summary = f"[历史摘要] {old_text[:200]}..."  # 实际应调用 LLM 摘要

    summary_msg = {"role": "system", "content": summary}
    return [summary_msg] + recent_messages
```

## 三、语义缓存（Semantic Cache）

80% 的用户问题落在 20% 的常见话题上。语义缓存不是精确匹配缓存键，而是对语义相似的查询返回相同的答案：

```python
import hashlib
import json
from sentence_transformers import SentenceTransformer
import numpy as np

class SemanticCache:
    def __init__(self, similarity_threshold: float = 0.92):
        self.model = SentenceTransformer("BAAI/bge-small-zh-v1.5")
        self.threshold = similarity_threshold
        self.cache: list[dict] = []  # [{"query": str, "embedding": np.ndarray, "answer": str}]

    def get(self, query: str) -> str | None:
        """查找语义相似的历史查询"""
        query_embedding = self.model.encode(query, normalize_embeddings=True)

        for entry in self.cache:
            similarity = np.dot(query_embedding, entry["embedding"])
            if similarity >= self.threshold:
                return entry["answer"]

        return None

    def set(self, query: str, answer: str):
        """缓存新查询的结果"""
        embedding = self.model.encode(query, normalize_embeddings=True)
        self.cache.append({
            "query": query,
            "embedding": embedding,
            "answer": answer
        })

# 使用示例
cache = SemanticCache()

queries = [
    "如何在 Python 中实现异步 HTTP 请求？",
    "Python 异步请求怎么做？",            # 语义类似 → 命中缓存
    "请用 aiohttp 写一个异步爬虫",        # 语义不同 → 需重新请求
]

for q in queries:
    cached = cache.get(q)
    if cached:
        print(f"[CACHE HIT]  {q[:30]}...")
    else:
        # 模拟 LLM 调用
        answer = f"回答: {q}"
        cache.set(q, answer)
        print(f"[LLM CALL]  {q[:30]}...")
```

**生产级语义缓存的三个关键点**：(1) 用轻量 Embedding 模型（如 bge-small）降低编码开销；(2) similarity_threshold 需要通过 A/B 测试校准，过高则命中率低，过低则答案不准确；(3) TTL 策略——FAQ 类查询缓存 7 天，事实性查询缓存 1 天，时效性查询不缓存。

## 四、模型分层路由

不是所有问题都需要 GPT-4。分层路由是用多个模型构建"漏斗"——简单问题用小模型（便宜），复杂问题用大模型（贵但准）：

```python
from dataclasses import dataclass
from typing import Literal

@dataclass
class ModelRouter:
    """模型分层路由器"""

    def classify_complexity(self, query: str) -> Literal["simple", "medium", "complex"]:
        """快速判断查询复杂度（用小模型做分类）"""
        # 规则 + 简单分类器的组合
        if len(query) < 20 and not any(kw in query for kw in ["分析", "对比", "总结"]):
            return "simple"
        if any(kw in query for kw in ["对比", "分析", "优化", "架构"]):
            return "complex"
        return "medium"

    def route(self, query: str):
        complexity = self.classify_complexity(query)
        model_map = {
            "simple": "gpt-4o-mini",      # $0.15/1M input
            "medium": "gpt-4o",           # $2.50/1M input
            "complex": "claude-sonnet-4"  # $3.00/1M input
        }
        selected_model = model_map[complexity]
        print(f"Query: {query[:40]}... → {complexity} → {selected_model}")
        return selected_model

router = ModelRouter()
queries = [
    "你好",                                      # → gpt-4o-mini
    "Python 列表推导式怎么写",                    # → gpt-4o
    "分析我们的微服务架构并提出优化方案"            # → claude-sonnet-4
]
for q in queries:
    router.route(q)
```

实际落地时，分类器应当用一个小模型（甚至规则引擎）+ 历史路由准确率 A/B 测试来持续优化路由策略。

## 五、成本监控与可视化

优化之前必须先测量。每个 Agent 请求应记录：

```python
import time
from dataclasses import dataclass, field

@dataclass
class TokenUsage:
    model: str
    input_tokens: int
    output_tokens: int
    cost: float
    latency_ms: float
    cache_hit: bool = False

class CostTracker:
    """按 model 和 session 维度的成本追踪器"""

    PRICING = {
        "gpt-4o":          (2.50, 10.00),   # $/1M input, output
        "gpt-4o-mini":     (0.15, 0.60),
        "claude-sonnet-4": (3.00, 15.00),
    }

    def __init__(self):
        self.records: list[TokenUsage] = []

    def record(self, usage: TokenUsage):
        self.records.append(usage)

    def calculate_cost(self, model: str, input_tokens: int, output_tokens: int) -> float:
        input_price, output_price = self.PRICING.get(model, (0, 0))
        return (input_tokens / 1_000_000) * input_price + \
               (output_tokens / 1_000_000) * output_price

    def report(self):
        total_cost = sum(r.cost for r in self.records)
        total_latency = sum(r.latency_ms for r in self.records)
        cache_hits = sum(1 for r in self.records if r.cache_hit)

        print(f"\n=== Token 消耗报告 ===")
        print(f"总请求: {len(self.records)}")
        print(f"缓存命中: {cache_hits} ({cache_hits/len(self.records)*100:.1f}%)")
        print(f"总成本: ${total_cost:.4f}")
        print(f"总延迟: {total_latency:.0f}ms")
        print(f"平均延迟: {total_latency/len(self.records):.0f}ms/请求")
```

## 六、最佳实践

1. **压缩率不要超过 50%**：LLMLingua 过度压缩会导致关键信息丢失，进而使 LLM 给出错误答案，反而增加重试成本。40%-50% 是生产验证的安全区间。

2. **语义缓存的 similarity_threshold 按场景校准**：FAQ 设 0.90-0.95；长文档问答设 0.85-0.88（表述差异大但意图一致）。

3. **分层路由需要观测回退率**：如果小模型处理复杂问题的比例 > 5%，说明分类器不准，应调高升级到中/大模型的阈值。

4. **缓存要区分时效性**：天气、股价等实时查询不缓存；百科知识缓存数天；代码错误修复缓存数周。

## 面试题

### 题1：滑动窗口和摘要窗口两种消息管理策略各有什么优缺点？在什么场景下应该选择摘要窗口？

**答案**：滑动窗口的优点是实现简单、无额外 LLM 调用成本、保留的消息是原始精确内容不会引入摘要误差；缺点是粗暴丢弃早期消息，可能丢失关键上下文——如用户在对话开头提到的约束条件（"预算 100 万以内"）如果在第 6 轮被丢弃，后续回答可能超出预算。

摘要窗口的优点是保留语义连续性，通过压缩而非丢弃来维持上下文凝练度，不会完全丢失早期信息；缺点是需要额外的 LLM 调用来生成摘要（增加延迟和成本），且摘要本身可能存在信息失真（摘要模型遗漏或曲解了关键细节）。

选择摘要窗口的场景：(1) 长时间多轮对话（>10 轮），用户可能在任意时刻回溯早期上下文；(2) 有约束条件的任务（法律咨询、需求分析），早期约束必须全程生效；(3) Agent 多次调用工具后积累了关键中间结果，不可简单丢弃。选择滑动窗口的场景：(1) 短对话（<5 轮）；(2) 每轮独立性强的问答（客服 FAQ）；(3) 对延迟和成本极度敏感的场景。

### 题2：语义缓存的相似度阈值如何选择？如果阈值设得太高或太低分别会有什么问题？

**答案**：相似度阈值决定了缓存的"宽容度"。阈值设得太高（如 0.98）→ 只有几乎逐字相同的查询才能命中缓存，命中率极低（可能 <5%），缓存形同虚设，每个查询仍走 LLM。阈值设得太低（如 0.80）→ 语义差异较大的查询也被判定为"相同"，返回的缓存答案可能不准确——例如"Python 中如何做单元测试"和"Python 中如何做集成测试"在 0.80 阈值下可能命中同一个缓存，但答案应该不同。

选择方法：(1) 准备一个评测集，包含语义相似度从高到低的 query pair，人工标注每对是否应该共享答案；(2) 在 0.85-0.95 区间内以 0.01 为步长遍历，找到 F1-score 最高的阈值；(3) 上线后监控缓存回退率（用户对缓存答案不满意而重新提问的比例），如果 > 5%，说明阈值过低需调高。生产经验值是 FAQ 场景 0.92-0.95，开放域问答 0.88-0.92。
