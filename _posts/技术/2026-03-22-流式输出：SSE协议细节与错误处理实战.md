## 一、核心概念

SSE（Server-Sent Events）是 AI Agent 应用中最常用的流式输出协议。它比 WebSocket 更轻量（单向推送、基于 HTTP）、比长轮询更高效，非常适合 LLM 逐个 Token 输出的场景。

SSE 的核心协议规范：

```
HTTP/1.1 200 OK
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive

event: message
data: {"delta": "你好", "index": 0}

event: message
data: {"delta": "，今天", "index": 1}

event: done
data: {"status": "completed", "total_tokens": 42}
```

每条消息格式：`field: value\n`，以空行 `\n` 分隔多条消息。支持四个标准字段——`data`（数据内容）、`event`（事件类型）、`id`（消息 ID，用于断线重连）、`retry`（重连间隔毫秒数）。

## 二、基础实现：FastAPI + 异步生成器

```python
from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse
import asyncio
import json
import time

app = FastAPI()

async def llm_stream_generator(query: str):
    """
    模拟 LLM 流式输出。
    生产环境中替换为实际的 LLM API 流式调用（OpenAI stream、Anthropic stream 等）。
    """
    response_text = (
        "LangGraph 是 LangChain 生态中的有向图状态机框架，"
        "通过节点和边来建模 Agent 的执行流程。"
    )

    for i, char in enumerate(response_text):
        # SSE 格式：data 字段包含 JSON
        chunk = json.dumps({"delta": char, "index": i}, ensure_ascii=False)
        yield f"data: {chunk}\n\n"
        await asyncio.sleep(0.03)  # 模拟逐字延迟

    # 结束事件，携带元信息
    done_msg = json.dumps({
        "status": "completed",
        "total_tokens": 84,
        "elapsed_ms": 1200
    }, ensure_ascii=False)
    yield f"event: done\ndata: {done_msg}\n\n"

@app.post("/chat/stream")
async def chat_stream(request: Request):
    """流式聊天端点"""
    body = await request.json()
    query = body.get("query", "")

    if not query:
        # 即使是错误，也用 SSE 格式返回（保持协议一致性）
        error_data = json.dumps({"error": "query 不能为空", "code": 400})
        return StreamingResponse(
            iter([f"event: error\ndata: {error_data}\n\n"]),
            media_type="text/event-stream",
            headers={
                "Cache-Control": "no-cache",
                "Connection": "keep-alive",
                "X-Accel-Buffering": "no",  # Nginx 禁用缓冲
            }
        )

    return StreamingResponse(
        llm_stream_generator(query),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",
            "Access-Control-Allow-Origin": "*",
        }
    )
```

## 三、进阶一：工具调用事件的流式处理

AI Agent 的核心交互是工具调用。流式输出中需要用 `event` 字段区分 LLM 的文本输出和工具调用：

```python
async def agent_stream_generator(query: str):
    """支持多事件类型的 Agent 流式输出"""
    # 阶段 1：LLM 思考中
    yield f"event: status\ndata: {json.dumps({'status': 'thinking'})}\n\n"

    # 阶段 2：LLM 逐字输出
    text = "我需要调用天气查询工具来获取当前天气信息。"
    for char in text:
        yield f"event: text\ndata: {json.dumps({'delta': char})}\n\n"
        await asyncio.sleep(0.01)

    # 阶段 3：工具调用开始
    yield f"event: tool_start\ndata: {json.dumps({'tool': 'get_weather', 'args': {'city': '北京'}})}\n\n"
    await asyncio.sleep(0.5)

    # 阶段 4：工具结果返回
    yield f"event: tool_result\ndata: {json.dumps({'tool': 'get_weather', 'result': '晴 22°C'})}\n\n"

    # 阶段 5：最终回答
    final = "根据查询结果，北京今天天气晴朗，气温22°C。"
    for char in final:
        yield f"event: text\ndata: {json.dumps({'delta': char})}\n\n"
        await asyncio.sleep(0.01)

    # 阶段 6：结束
    yield f"event: done\ndata: {json.dumps({'status': 'completed'})}\n\n"
```

**前端消费示例**（JavaScript EventSource）：

```javascript
const eventSource = new EventSource("/chat/stream");

// 文本逐字追加
eventSource.addEventListener("text", (e) => {

    document.getElementById("output").innerText += delta;

// 工具调用可视化

eventSource.addEventListener("tool_result", (e) => {
    const { tool, result } = JSON.parse(e.data);
    hideToolIndicator();
    showToolResult(tool, result);
});

eventSource.addEventListener("done", () => {
    eventSource.close();
});
```

## 四、进阶二：错误处理与断线重连

### 4.1 服务端异常捕获

流式生成器中的异常不会自动转换为 HTTP 错误响应——因为响应头已经发送了。必须在生成器内部捕获异常并通过 SSE 的 error 事件通知客户端：

```python
async def robust_stream_generator(query: str):
    """带异常捕获的流式生成器"""
    try:
        # 执行 Agent 流程
        async for event in agent_core.process(query):
            yield event

        # 正常结束
        done_msg = json.dumps({"status": "completed"})
        yield f"event: done\ndata: {done_msg}\n\n"

    except TimeoutError:
        error_msg = json.dumps({
            "error": "Agent 执行超时",
            "code": "TIMEOUT",
            "suggestion": "请缩小查询范围后重试"
        })
        yield f"event: error\ndata: {error_msg}\n\n"

    except Exception as e:
        error_msg = json.dumps({
            "error": str(e),
            "code": "INTERNAL_ERROR",
            "suggestion": "请稍后重试"
        })
        yield f"event: error\ndata: {error_msg}\n\n"
```

### 4.2 客户端断线重连

SSE 原生支持通过 `Last-Event-ID` 头实现断线重连——客户端断开后重连时，带上最后收到的 `id`，服务端从该位置之后继续推送：

```python
@app.post("/chat/stream")
async def chat_stream_with_reconnect(request: Request):
    body = await request.json()
    query = body.get("query", "")
    last_event_id = request.headers.get("Last-Event-ID", "0")

    return StreamingResponse(
        agent_stream_with_id(query, int(last_event_id)),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "Connection": "keep-alive",
            "X-Accel-Buffering": "no",
        }
    )

async def agent_stream_with_id(query: str, last_id: int):
    """支持断线续传的流式生成器"""
    event_id = last_id

    async for event in agent_core.process(query):
        event_id += 1
        # SSE 的 id 字段 → 客户端断开后会用 Last-Event-ID 带上这个值
        yield f"id: {event_id}\n{event}"
```

## 五、进阶三：心跳保活与超时控制

Nginx 和某些 CDN 会在无数据推送超过 60 秒时断开连接。心跳机制解决此问题：

```python
async def stream_with_heartbeat(generator, heartbeat_interval: float = 15.0):
    """在流式输出中插入心跳注释"""
    gen = generator.__aiter__()

    while True:
        try:
            # 带超时的获取下一个事件
            chunk = await asyncio.wait_for(gen.__anext__(), timeout=heartbeat_interval)
            yield chunk
        except asyncio.TimeoutError:
            # 超时 → 发送心跳（SSE 中以冒号开头的行是注释，客户端忽略）
            yield ": heartbeat\n\n"
        except StopAsyncIteration:
            break
```

## 六、最佳实践

1. **用 `event` 字段做语义分层**：不要把所有数据都塞在 `data` 里用 JSON 的 `type` 字段区分。SSE 原生的 `event` 字段让客户端可以用 `addEventListener` 直接订阅特定类型，代码更清晰。

2. **`X-Accel-Buffering: no` 是必选项**：Nginx 默认缓冲 SSE 响应，导致客户端收不到逐字推送。这个头告诉 Nginx 不要缓冲。

3. **错误也要走 SSE 协议**：不要在生成器内部直接抛异常让 HTTP 框架转成 500 JSON。始终通过 `event: error` 通知客户端，保持协议一致性。

4. **前端 EventSource 不支持自定义 Header**：如需认证，用 URL 参数（如 `?token=xxx`）或改用 `fetch` + `ReadableStream` 手动解析 SSE。

5. **连接数控制**：SSE 是长连接，每个连接占用一个线程/协程。生产环境需用 uvicorn + gunicorn 的多 worker 模式，配合连接数监控和限流。

## 面试题

### 题1：SSE 和 WebSocket 的核心区别是什么？为什么 AI Agent 的流式输出场景更适合用 SSE 而非 WebSocket？

**答案**：核心区别在于通信方向——SSE 是**单向**的（Server → Client），基于标准 HTTP 协议，客户端只需一个简单的 EventSource 即可接收；WebSocket 是**双向**的，需要协议升级握手（HTTP Upgrade），客户端和服务端都需要专门的库。

AI Agent 场景更适合 SSE 的原因：(1) Agent 的通信模式天然是"请求→流式响应"，用户发送一次 query，服务端持续推送结果，不需要双向实时通信；(2) SSE 基于 HTTP，天然享受 HTTP 生态的所有基础设施——负载均衡、代理缓存、认证中间件无需额外适配；(3) 实现简单——FastAPI 十几行代码即可构建 SSE 端点，WebSocket 需要管理连接池、心跳、重连逻辑；(4) SSE 原生支持 `Last-Event-ID` 断线重连，而 WebSocket 需要自己实现这个机制。

WebSocket 更适用的场景：需要客户端频繁向服务端发送消息的交互（如多人协作白板、实时游戏），或需要<毫秒级双向低延迟的场景。

### 题2：流式输出中如果 LLM 调用中途抛出异常（如 API 超时），应该如何优雅地通知客户端？为什么不能在生成器内部直接 `raise HTTPException`？

**答案**：不能在生成器内部 `raise HTTPException` 的原因：流式响应的 HTTP 状态码和响应头在第一个 `yield` 之前就已经发送给客户端了（200 OK + `Content-Type: text/event-stream`）。此时再抛异常，FastAPI/Starlette 尝试修改状态码为 500，但 HTTP 协议不允许在响应头发送后再修改——客户端已经认为请求成功，但收到的数据流突然中断，表现为前端 EventSource 触发 `onerror` 但拿不到任何错误信息。

正确的做法是捕获异常后，通过 SSE 协议的 `event: error` 消息将错误信息推送给客户端：(1) 在生成器内部用 `try/except` 包裹 Agent 执行逻辑；(2) 捕获异常后构造结构化的错误 JSON（包含错误码、描述、建议操作）；(3) 通过 `yield` 以 SSE 标准格式发送；(4) 发送完后正常关闭生成器（`return`），让连接优雅断开。前端 EventSource 监听 `error` 事件即可获取完整的错误上下文，并根据 `code` 字段决定是否重试。
